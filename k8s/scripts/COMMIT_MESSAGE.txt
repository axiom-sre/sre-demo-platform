## Root cause fixes

### boutique/boutique.yaml → v5

**avoidNoopCurrencyConversionRPC=true (frontend rebuild)**
- Problem: homepage called currencyservice 11× per request (1 + 10 per product)
- Fix: rebuilt frontend with avoidNoopCurrencyConversionRPC=true
- Result: 10× reduction in currency RPC traffic
- Image: localhost:5001/boutique-frontend:arm64-v1 (arm64, local registry)
- Build: /tmp/boutique-patch/src/frontend (sparse clone, one-line patch)

**minReadySeconds on productcatalogservice (10s) and recommendationservice (20s)**
- Problem: 4 HPA scale events in 4 minutes at 300→600VU ramp caused 491 errors
- Root cause: tcpSocket readiness passes when port opens (~5s) but service
  initialization takes 10-20s longer. k8s routed traffic to warming pods.
- Fix: minReadySeconds gates Available state until service is truly ready
- Confirmed via kubectl events: recommendationservice 2→3→4→5 in 3 minutes

**terminationGracePeriodSeconds: 30 on recommendationservice**
- Python process needs drain time for in-flight gRPC requests on scale-down
- Made explicit (was default 30s, now documented and intentional)

### scripts/load-test_1000vusers.js
- maxConnectionsPerHost moved to comment — deprecated in k6 v0.44
- k6 handles connection pooling automatically at 1000VU
- Removes WARN[0000] noise from test output
